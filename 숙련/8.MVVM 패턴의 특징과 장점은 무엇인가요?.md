# MVVM의 특징
데이터 바인딩(Data Binding): ViewModel의 상태가 변하면 View가 이를 자동으로 감지하여 업데이트합니다. Flutter에서는 ref.watch()가 이 역할을 수행합니다.

의존성 분리: View는 ViewModel을 알지만, ViewModel은 View를 참조하지 않습니다. 이 덕분에 UI를 아무리 수정해도 로직에는 영향이 없습니다.

상태 중심: UI의 변경을 직접 명령(Imperative)하는 것이 아니라, 상태(State)를 변경하면 UI가 그에 반응(Reactive)하도록 설계합니다.
# MVVM의 장점

1) 유지보수성 (Maintainability)
UI 디자인이 대폭 수정되어도 ViewModel의 로직은 건드릴 필요가 없습니다. 반대로 로직이 바뀌어도 UI 코드는 그대로 유지될 수 있어 관리가 매우 수월합니다.

2) 테스트 용이성 (Testability)
UI(View)는 테스트하기 까다롭지만, ViewModel은 순수하게 데이터와 로직만 담고 있는 클래스입니다. 따라서 시뮬레이터나 실제 기기 없이도 로직이 정상 작동하는지 **단위 테스트(Unit Test)**를 수행하기 매우 좋습니다.

3) 코드 재사용성
하나의 ViewModel을 여러 개의 View에서 공유할 수 있습니다. 예를 들어, 유저의 프로필 정보를 담은 ViewModel 하나를 '마이페이지'와 '설정 페이지'에서 동시에 구독하여 사용할 수 있습니다.

4) 협업 효율성
디자이너와 프론트엔드 개발자가 UI(View)를 작업하는 동안, 백엔드 로직을 다루는 개발자는 ViewModel과 Model을 독립적으로 개발할 수 있어 병렬 작업이 가능해집니다.
