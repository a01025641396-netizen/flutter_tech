RESTful API의 정의 및 핵심 원칙
RESTful API란 REST(Representational State Transfer)라는 아키텍처 스타일의 제약 조건을 엄격히 준수하여 설계된 API를 의미합니다. 단순히 데이터를 주고받는 것을 넘어, 자원(Resource)을 이름으로 구분하여 해당 자원의 상태를 주고받는 통신 방식입니다.

1. REST의 3가지 핵심 요소
RESTful API는 다음 세 가지 요소로 명확하게 구성됩니다.

자원 (Resource): 모든 자원은 고유한 URI(Uniform Resource Identifier)를 가집니다. (예: /users/1)

행위 (Verb): 자원에 대한 조작은 HTTP Method를 통해 표현합니다. (GET, POST, PUT, DELETE)

표현 (Representation): 자원의 상태는 JSON, XML 등의 형태로 전달됩니다.

2. REST의 6가지 핵심 설계 원칙
RESTful하다고 평가받기 위해서는 다음의 원칙들을 준수해야 합니다.

Uniform Interface (일관된 인터페이스): URI로 지정된 자원에 대한 조작이 통일되고 한정적인 인터페이스로 수행되어야 합니다. 이는 특정 언어나 플랫폼에 종속되지 않게 합니다.

Stateless (무상태성): 서버는 클라이언트의 상태를 저장하지 않습니다. 각 요청은 처리에 필요한 모든 정보를 포함해야 하며, 이는 서버의 확장성을 높여줍니다.

Caching (캐시 가능성): HTTP의 기존 웹 표준을 그대로 사용하므로, 캐싱 기능을 적용하여 대량의 요청을 효율적으로 처리할 수 있습니다.

Client-Server Architecture (클라이언트-서버 구조): 자원을 가진 쪽이 서버, 요청하는 쪽이 클라이언트로 명확히 구분되어 서로 간의 의존성을 낮춥니다.

Hierarchical System (계층화 시스템): 클라이언트는 서버의 구성과 무관하게 통신합니다. 서버는 보안, 로드 밸런싱, 공유 캐시 등 다중 계층으로 구성될 수 있습니다.

Code on Demand (선택적 사항): 서버에서 스크립트를 클라이언트로 보내 실행할 수 있게 합니다.

3. RESTful API 설계의 관례 (Best Practice)
명사 사용: URI에는 행위(동사)가 아닌 자원(명사)을 사용합니다.

(나쁜 예) /getUsers/1

(좋은 예) /users/1

HTTP Method의 활용: 행위는 URI가 아닌 Method로 구분합니다.

GET: 자원 조회

POST: 자원 생성

PUT: 자원 전체 수정

PATCH: 자원 일부 수정

DELETE: 자원 삭제
